漏斗分析（Funnel Analysis）係產品分析入面最基礎亦都最實用嘅方法之一。佢追蹤用戶喺一連串有序步驟入面嘅行為，量化每一步嘅轉換同流失。呢篇文用 HelloFresh 嘅訂閱制商業模式做例，由定義漏斗、計算指標到搵出瓶頸同優化策略，完整拆解呢套分析框架。

> 注意：文中所有數據都係假設，淨係用嚟說明分析方法。

## 咩係漏斗分析？

漏斗分析係一種**序列模型**，追蹤用戶由第一個接觸點到最終目標（好似購買、註冊）之間，每一步嘅完成率同流失率。

核心概念：

- **轉換率（Conversion Rate）**：完成當前步驟並進入下一步嘅用戶比例
- **流失率（Drop-off Rate）**：喺當前步驟離開、無進入下一步嘅比例
- **整體轉換率**：由漏斗頂端到底端嘅端到端比例

同其他分析方法嘅分別：

| 方法 | 回答嘅問題 |
|------|-----------|
| 漏斗分析 | 用戶喺邊一步流失最多？ |
| 世代分析（Cohort） | 唔同批次嘅用戶行為有咩分別？ |
| 留存分析（Retention） | 用戶係咪持續返嚟？ |

三者通常一齊用，但漏斗分析係診斷轉換問題嘅第一步。

## HelloFresh 嘅業務背景

HelloFresh 係全球最大嘅 meal-kit 訂閱服務，商業模式好清晰：用戶喺網上揀餐盒方案，每週收到食材同食譜，喺屋企煮嘢食。

呢個模型好適合用嚟示範漏斗分析，因為：

1. **路徑明確**：由瀏覽到訂閱有好清楚嘅線性流程
2. **訂閱制**：唔止睇首次購買，仲要睇續訂同留存
3. **多個摩擦點**：揀方案、註冊、自訂菜單、結帳，每一步都可能流失

關鍵商業指標：

- **CAC（Customer Acquisition Cost）**：攞一個新客嘅成本
- **Trial-to-Paid Conversion**：試用轉正式訂閱嘅比率
- **Churn Rate**：每月取消訂閱嘅比率

## 定義漏斗步驟

一個典型嘅 HelloFresh 新用戶漏斗可以咁定義：

| 步驟 | 事件名稱 | 說明 |
|------|---------|------|
| Step 1 | `page_view_landing` | 訪問首頁或活動頁 |
| Step 2 | `plan_browse` | 瀏覽方案頁面 |
| Step 3 | `plan_select` | 揀定一個方案 |
| Step 4 | `signup_complete` | 完成註冊或登入 |
| Step 5 | `menu_customize` | 自訂本週菜單 |
| Step 6 | `checkout_complete` | 完成結帳付款 |
| Step 7 | `reorder` | 第二週之後嘅續訂 |

### 粒度嘅取捨

漏斗步驟唔係越多越好：

- **太少**（例如淨係有「訪問 → 結帳」）：睇唔出問題喺邊
- **太多**（例如拆到每個表單欄位）：噪音太大，難以聚焦
- **建議**：3-7 個步驟，每步對應一個有意義嘅用戶決策點

## 計算轉換率同流失率

假設我哋追蹤咗一個月嘅數據：

| 步驟 | 用戶數 | 步進轉換率 | 累計轉換率 |
|------|--------|-----------|-----------|
| 首頁訪問 | 100,000 | — | 100% |
| 瀏覽方案 | 60,000 | 60.0% | 60.0% |
| 選擇方案 | 25,000 | 41.7% | 25.0% |
| 註冊 | 15,000 | 60.0% | 15.0% |
| 自訂菜單 | 10,000 | 66.7% | 10.0% |
| 結帳 | 7,000 | 70.0% | 7.0% |
| 續訂 | 3,500 | 50.0% | 3.5% |

公式：

```
步進轉換率 = 當步用戶數 / 上一步用戶數
累計轉換率 = 當步用戶數 / 第一步用戶數
流失率 = 1 - 步進轉換率
```

由呢張表可以快速睇到：**瀏覽方案 → 選擇方案**（41.7%）同**結帳 → 續訂**（50.0%）係兩個最大嘅瓶頸。

## 搵出瓶頸：常見分析手法

淨係睇整體漏斗唔夠，你要切維度嚟診斷根因。

### 1. 逐步 Drop-off 分析

搵出 drop-off 最大嗰步，優先處理。上面個例入面「瀏覽方案 → 選擇方案」流失咗 58.3% 嘅用戶，係最應該優先調查嘅環節。

### 2. 按渠道分群

| 渠道 | 首頁 → 結帳轉換率 |
|------|-------------------|
| Google Ads | 5.2% |
| 自然搜尋 | 8.1% |
| 朋友推薦 | 12.3% |
| 社交媒體 | 3.8% |

朋友推薦嘅轉換率遠高過付費廣告，可能代表推薦計畫嘅 ROI 更高。

### 3. 按裝置分群

| 裝置 | 自訂菜單 → 結帳轉換率 |
|------|----------------------|
| 桌面 | 78.5% |
| 手機 | 58.2% |

手機端結帳轉換率明顯偏低，可能係行動端結帳流程有 UX 問題。

### 4. Time-to-Convert 分析

用戶喺每一步停留幾耐先進入下一步？

- 瀏覽方案 → 選擇方案：中位數 4.2 分鐘（諗好耐，可能方案比較太複雜）
- 自訂菜單 → 結帳：中位數 1.1 分鐘（順暢，無明顯摩擦）

停留時間太長通常代表用戶遇到困惑或者猶豫。

## 優化策略

針對每個瓶頸，用唔同嘅優化手段：

### 瀏覽 → 選擇（41.7%，最大瓶頸）

- **簡化方案比較**：減少方案數量，突出差異，加推薦標籤（好似「最受歡迎」）
- **A/B 測試定價頁**：試唔同排版、預設選項、價格錨定
- **加社會證明**：顯示訂閱人數、評分、用戶評價

### 選擇 → 註冊（60.0%）

- **降低註冊摩擦**：支援 Google/Apple 社交登入
- **延遲註冊**：先畀用戶揀完菜單先要求註冊（先製造沉沒成本）
- **訪客結帳**：允許唔註冊直接買

### 自訂菜單 → 結帳（70.0%）

- **預設菜單**：提供系統推薦嘅默認搭配，減少選擇負擔
- **儲存偏好**：記住飲食偏好（素食、無麩質等），自動篩選

### 結帳 → 續訂（50.0%，留存關鍵）

- **Onboarding 郵件**：第一週寄烹飪技巧、食譜影片
- **難度配對**：根據用戶經驗推薦適合難度嘅食譜
- **彈性暫停**：允許跳過一週而唔係直接取消
- **離開挽留**：取消流程中提供折扣或方案調整

### 優先排序：Impact × Effort

| 優化項目 | 預期影響 | 實作難度 | 優先序 |
|---------|---------|---------|-------|
| 簡化方案比較 | 高 | 低 | P0 |
| 社交登入 | 中 | 低 | P1 |
| 預設菜單 | 中 | 中 | P1 |
| Onboarding 郵件 | 高 | 中 | P0 |
| 彈性暫停 | 高 | 高 | P2 |

## 工具同實作

### 分析平台

- **Mixpanel / Amplitude**：內建漏斗分析功能，拖拉設定步驟就可以出報表
- **Google Analytics 4**：漏斗探索報告（Exploration → Funnel）
- **自建方案**：事件表 + SQL 查詢

### SQL 範例

假設你有一張事件表 `events(user_id, event_name, timestamp)`，用視窗函數計算漏斗：

```sql
WITH funnel AS (
  SELECT
    user_id,
    MAX(CASE WHEN event_name = 'page_view_landing' THEN 1 ELSE 0 END) AS step1,
    MAX(CASE WHEN event_name = 'plan_browse' THEN 1 ELSE 0 END) AS step2,
    MAX(CASE WHEN event_name = 'plan_select' THEN 1 ELSE 0 END) AS step3,
    MAX(CASE WHEN event_name = 'signup_complete' THEN 1 ELSE 0 END) AS step4,
    MAX(CASE WHEN event_name = 'checkout_complete' THEN 1 ELSE 0 END) AS step5
  FROM events
  WHERE timestamp BETWEEN '2026-02-01' AND '2026-02-28'
  GROUP BY user_id
)
SELECT
  COUNT(*) AS total_users,
  SUM(step1) AS landing,
  SUM(CASE WHEN step1 = 1 AND step2 = 1 THEN 1 ELSE 0 END) AS browse,
  SUM(CASE WHEN step1 = 1 AND step2 = 1 AND step3 = 1 THEN 1 ELSE 0 END) AS selected,
  SUM(CASE WHEN step1 = 1 AND step2 = 1 AND step3 = 1 AND step4 = 1 THEN 1 ELSE 0 END) AS signup,
  SUM(CASE WHEN step1 = 1 AND step2 = 1 AND step3 = 1 AND step4 = 1 AND step5 = 1 THEN 1 ELSE 0 END) AS checkout
FROM funnel;
```

呢段 SQL 確保用戶必須**依序完成**每一步先被計入下一步，避免跳步造成嘅數據膨脹。

### 呈現畀 Stakeholder

- 用**水平柱狀圖**呈現漏斗，每一步標註轉換率同絕對數字
- 加**趨勢折線圖**，追蹤每週漏斗轉換率嘅變化
- 附**分群比較表**，畀產品經理一眼睇出邊個渠道或裝置需要優化

## 常見陷阱

### 1. 只睇虛榮指標

只優化漏斗頂端（好似首頁流量）但忽略下游轉換，最終唔會帶嚟營收增長。

### 2. 歸因窗口唔清

用戶今日瀏覽、下週先結帳，算唔算同一個漏斗？你需要定義清楚嘅**歸因窗口**（例如 7 日內完成所有步驟先計入）。

### 3. 生存者偏差

只分析進入漏斗嘅用戶，忽略咗完全無進入漏斗嘅人。可能最大嘅問題唔係喺漏斗入面，而係根本無人知你個產品。

### 4. 過度切分

將數據切成太多維度（渠道 × 裝置 × 地區 × 新舊客），樣本量太細，結論唔可靠。切分前先確認每個分群有足夠嘅樣本。

### 5. 只睇數字唔睇原因

漏斗分析話你知**邊度**有問題，但唔話你知**點解**。配合質化方法（用戶訪談、Session Recording、問卷）先搵到真正嘅根因。

## 結語

漏斗分析係產品分析嘅起點，唔係終點。佢幫你快速定位轉換瓶頸，但真正嘅優化需要結合分群分析、A/B 測試同質化研究。

用 HelloFresh 做例，我哋可以睇到：即使係一個睇落好簡單嘅「瀏覽→購買」流程，拆解之後都藏住好多可優化嘅環節。關鍵係：定義清楚嘅步驟、追蹤啱嘅事件、用數據搵出瓶頸、然後用 Impact × Effort 排序嚟決定先做咩。

記住：**漏斗會跟住產品演進而改變**。定期重新檢視漏斗定義，確保佢反映嘅係而家嘅用戶旅程，唔係半年前嘅假設。
